## 二、动机

[zh_02_dongji.mp3](../assets/audio/zditor/zh/zh_02_dongji.mp3|mode=audio)

我最早的时候做笔记文档啥的用的印象笔记，然后其他平台也都用过，后来发现不好用，由于数据迁移困难，最后数据都烂在各平台不要了。后来写文档做笔记啥的都建立一个github的仓库，然后本地用typora或者vscode编辑。再后来，我自己需要分享一些文档给别人，调研发现虽然有很多项目支持直接同步仓库，比如gitbook，mintlify,但是这些项目通常要么只能公开，要么只能私有，更细粒度的权限是付费功能。我的文档比如公司相关的显然不能公开，有些笔记自己写的可以分享给别人看。所以我就做了一个项目zbook，一个多租户的markdown文档平台，支持多用户在网页查看渲染好的markdown文档。zbook没有文档编辑功能，然后我就做了zditor，一个本地的markdown编辑器。

## 三、调研编辑器软件

[zh_03_diaoyan_ruanjian.mp3](../assets/audio/zditor/zh/zh_03_diaoyan_ruanjian.mp3|mode=audio)

经过一番调研，可以将笔记编辑器软件分成三类，一类是云端的，比如notion，一类是本地数据库的，比如affine，另一类是本地基于文件的，比如typora，obsidian，vscode。

你可以给基于云端的产品找一堆毛病，比如隐私，比如网络、比如数据迁移。但是对我来讲核心问题是这些产品都是富文本编辑器，对markdown的支持不够友好，独特的格式对用户是一种负担，数据又会烂在里面。另一类本地基于数据库的其实也是这个问题。所以毫无疑问，本地基于文件才会是我的选择。

## 四、调研编辑器本身

[zh_04_diaoyan_bensheng.mp3](../assets/audio/zditor/zh/zh_04_diaoyan_bensheng.mp3|mode=audio)

再次调研编辑器工具本身，有很多框架，流行的，比如prosemirror，比如基于prosemirror封装的tiptap，我真的很喜欢tiptap，但是当年tiptap对markdown的支持比较有限，性能拉垮，也为了更好的满足自己的格式需要做定制的插件，所以决定类似tiptap，在prosemirror上封装自己的编辑器。

本来想着已经有大量的prosemirror插件了，还有那么多的开源库可以借鉴，还有AI辅助不会太难，然而实际上这个坑比我想的要深的多。比如跨平台是一个很大的问题，尤其是我使用的tauri框架，导致光标、选区、在不同平台的差异导致了一堆又一堆的bug需要修复。

## 五、使用场景

[zh_05_shiyong_changjing.mp3](../assets/audio/zditor/zh/zh_05_shiyong_changjing.mp3|mode=audio)

下面讲下zditor的一些使用场景。本来是想做一个类似typora的产品，聚焦编辑器相关功能，所见即所得，用户在软件中编辑，更改实时同步落盘到文件。不过后面逐渐追加功能，扩展使用场景，比如文档的修订和注释，本地notebooklm，本地Agent等场景。

### 场景 1：文字与画布创作

首先讲文档的修订和注释，这个主要想着是比如你写一个文章，里面可能有错别字，然后让ai用修订语法给你标注出来：这个错了，为啥错，应该改成啥，然后你选择是否接受修订，还是要自己特殊处理。

### 场景 2：本地版 NotebookLM

然后讲本地版本notebooklm，实际上这个就是调用电脑上安装好的agent来实现。下载一个代码仓库，借助 zditor 快速生成学习文档、PPT、甚至视频。相比 NotebookLM 有三个优势：首先是**更快**：可以并行生成多张图；可以**编辑**：生成的内容随时按你的意图修改；**效果更好**：claude + nano banana 的组合优于单独用 gemini。

### 场景 3：本地 Agent 入口

zditor 接入先进 Agent——直连 claude code、gemini cli、codex、openclaw，你可以在文档编辑器里获得和代码 Agent 同等级别的智能能力。

## 六、踩过的坑：Tauri 选型

[zh_06_tauri_keng.mp3](../assets/audio/zditor/zh/zh_06_tauri_keng.mp3|mode=audio)

最开始技术选型时，我对比了 electron 和 tauri，被 tauri 的几个卖点打动——包体积小、rust 内存安全、启动快——就选了 tauri。

事后回头看，**这是整个项目最大的一次决策失误**，代价非常显性。

1. **开发效率至少打 5 折**。同样功能，tauri 上的开发难度至少是 electron 的两倍——社区生态、文档、可复用包、踩坑记录都远不如 electron。
2. **兼容性最崩溃**。tauri 依赖系统 webview——macOS 是 webkit、windows 是 webview2、linux 是 webkitgtk，行为差异大。同 OS 不同版本能力也不一样，3 年前的系统可能因为缺 css/js 特性就得专门 patch。编辑器对光标、选区、中文输入法、contenteditable 极其敏感，safari/webkit 上踩了一堆坑：选区跳变、输入法候选框错位、组合输入丢字……chromium 上基本不存在。
3. **UI 一致性难保证**。同一份前端代码，macOS 好好的，windows 上字体、滚动、动画各显神通。
4. **linux 几乎不可用**。webkitgtk 版本碎片化严重，发行版之间还会放大差异。
5. **性能并不省心**。冷启动和包体积小是真的，但运行时性能受限于系统 webview——safari/webkit 下大文档渲染、prosemirror 事务处理性能只有 chromium 的一半左右，反而成了瓶颈。
6. **调试链路更长**。一个 bug 可能来自前端、rust 后端、IPC 桥接、系统 webview 任意一层。排查成本明显高于 electron 的清晰结构。

为啥不迁移到electron呢，tauri本身有很多优点，比如细粒度的权限控制，相比之下electron的obsidian的第三方插件几乎能获得所有权限，这个应该是很危险的事情。此外已经有很多逻辑在rust侧了，迁移成本太高，另一方面在等tauri出cef版本，如果出了话，上面的大部分问题都可以解决了。

## 七、路线图

[zh_07_luxiantu.mp3](../assets/audio/zditor/zh/zh_07_luxiantu.mp3|mode=audio)

接下来想做的事，首先是excalidraw画布，支持更多的markdown节点类型：希望从 markdown 拖一个元素到画布时自动创建对应类型节点（比如音频、视频、附件等）。然后要支持画布的节点工具栏，要像 markdown 工具栏一样把 AI 能力扩展进来——节点转手绘风格、格式化等。然后支持套索工具栏：选中部分节点后做批量操作最后要支持利用ai实现**markdown到画布思维导图转换**：希望节点对节点级别的转换，目前的实现还比较简陋。

除了画布之外，还打算做移动端，其实tauri本身就支持移动端，所以这里的工作量要少一些，主要是ui的适配，比如编辑器工具栏可能要放在键盘上方；不支持的功能（如拖拽）要删掉。另一点是同步：移动端系统封闭，没有同步用起来很麻烦。具体方案还没想好，欢迎评论区留言。

## 八、插件化

[zh_08_chajiianhua.mp3](../assets/audio/zditor/zh/zh_08_chajiianhua.mp3|mode=audio)

如果上面都做好了，后续可能会做插件化。有人说插件是 obsidian 的护城河，但在我看来这件事并不难做。obsidian的90% 的插件其实没多少人用，此外在AI 时代这 90% 的插件交给 AI 写两小时内应该都能搞定。少量高质量插件可以做成官方的。AI 时代生态真正的壁垒，是那些少量、高质量的插件。

## 九、商业化

[zh_09_shangyelhua.mp3](../assets/audio/zditor/zh/zh_09_shangyelhua.mp3|mode=audio)

软件的本质是规模化——边际成本趋近于零，只有用户量级起来，前期投入才有意义。反过来说：目标用户群本身很小就没必要投入过多资源；所在领域平替很多，不做到最优秀就无法扩大用户群，回本无从谈起。

回到 zditor，过去一个多月获得了一些用户付费支持，但这远远不够——现在的成本仅 token 一项已经过万，更别提背后几千小时的人工成本。卖出几十、几百份授权根本不够回本。所以**出海获取更多用户是当务之急**，也是一开始就设想好的路。这里也可以聊一聊很多独立开发者做的竞品，大部分都是套下编辑器的壳子，然后做一下UI，而且主要都是走本地数据库路线的，这个方式在我看来做不大的话几乎都没啥希望，比如只一两万用户，付费了2%，也就几百付费用户，最后不维护了，怕是用户数据最后还是烂软件里面，所以这也是我要支持markdown，要支持excalidraw，而不是自己做画布的原因：即使最终zditor失败了，用户也可以轻松迁移数据。更何况因为本身就是markdown编辑器，双击markdown文件仍然可以正常用

### 商业模式的演进路径

我对 zditor 的商业化思路是：**分阶段演进，尊重老用户**。第一阶段，买断制，也就是现在这个阶段，愿不愿意为买断付费，是验证用户真实意愿最干净的信号，比任何问卷、免费试用都直接。这个阶段如果不能卖出去几百份，估计这个软件就要死掉了。第二阶段，订阅制（面向新用户），等产品成熟度和更新节奏稳定后，会切换为订阅制，匹配持续投入的成本结构。**老用户老办法、新用户新办法**：已买断用户不会被强制转订阅，也不会被阉割成订阅版本的降级形态。后续还打算出AI服务，同步服务等，如果能走到这一步的话，打算给予早期买断制用户9折的优惠。

## 十、总结

[zh_10_zongjie.mp3](../assets/audio/zditor/zh/zh_10_zongjie.mp3|mode=audio)

总的想法是尽可能做到最好，比如会将UI做的更好，现在的ui完全不满足我的要求，但是仔细调整需要的精力太多了，从50分到80分很容易，而从80分到100分是很难的。从我的角度来看，可以在1-2年内满足50%-80%的obsidian的目标用户的需求，这样目标用户的规模也有百万级了。从技术上讲超越obsidian不是啥难事，但是商业上讲可能走不到那一天。过去这些年做了好多个项目，从开始的做的端侧AI模型，到蓝牙聊天，再到zbook，虽然每一个都花了很多心力，最后都失败了。不知道这次能否走的更远了，无论最后结果会如何，还是感谢大家的支持。
