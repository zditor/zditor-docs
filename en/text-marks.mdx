Text marks are one of the core capabilities of the Zditor Markdown editor. They control inline formatting such as emphasis, highlight, links, inline code, and review marks.

## What are text marks?

Marks are formatting properties applied to a range of inline text. Unlike block elements such as headings or paragraphs, marks can be attached to any span of text inside a paragraph. Zditor supports seven main mark types.

## Basic text marks

### Bold

Use bold for important terms and key conclusions.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Wrap text with double asterisks:

    ```markdown
    This is **bold text**.
    ```
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + B`
    - Windows/Linux: `Ctrl + B`
  </Accordion>

  <Accordion title="Auto-conversion" icon="sparkles">
    Typing `**text**` and then pressing space or enter converts it into a bold mark.
  </Accordion>
</AccordionGroup>

<Tip>
  Use bold sparingly. If everything is emphasized, nothing is emphasized.
</Tip>

### Italic

Italic is useful for mild emphasis, foreign words, or special terms.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Wrap text with `_text_` or `*text*`.
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + I`
    - Windows/Linux: `Ctrl + I`
  </Accordion>

  <Accordion title="Auto-conversion rules" icon="sparkles">
    Italic conversion is intentionally stricter than bold so identifiers such as `snake_case` do not convert accidentally.

    **Usually converts**
    - `_123_`
    - `(_one_`
    - `[_one_`

    **Usually does not convert**
    - `__`
    - `one_two`
    - `ONE_TWO`
  </Accordion>
</AccordionGroup>

<Warning>
  In technical writing, prefer `*asterisk*` syntax for italic if your content includes many underscore-based identifiers.
</Warning>

### Strikethrough

Use strikethrough for removed text, completed items, or before-and-after changes.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Wrap text with `~text~` or `~~text~~`.
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + Shift + X`
    - Windows/Linux: `Ctrl + Shift + X`
  </Accordion>

  <Accordion title="Common uses" icon="lightbulb">
    - Show removed content
    - Mark completed items
    - Show original pricing or replaced wording
  </Accordion>
</AccordionGroup>

### Highlight

Use highlight for definitions, takeaways, or reading notes that need extra visibility.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Wrap text with `==text==`.
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + Shift + H`
    - Windows/Linux: `Ctrl + Shift + H`
  </Accordion>

  <Accordion title="Styling" icon="palette">
    Highlight uses a yellow background by default and can support custom color styling through editor configuration.
  </Accordion>
</AccordionGroup>

## Functional marks

### Inline code

Inline code is for commands, variables, identifiers, short snippets, and file names.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Wrap the content in backticks:

    ```markdown
    Run `npm install`.
    Use the variable `userName`.
    ```
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + E`
    - Windows/Linux: `Ctrl + E`
  </Accordion>

  <Accordion title="Smart input" icon="wand-sparkles">
    Typing `` `code` `` and pressing space converts it automatically. When the cursor is between matching backticks, typed or pasted content stays inside the inline code mark.
  </Accordion>

  <Accordion title="Exclusion rule" icon="ban">
    Inline code is mutually exclusive with other marks. If you apply code formatting, Zditor removes other inline marks from that range so the text renders as plain code.
  </Accordion>
</AccordionGroup>

<Note>
  Inline code uses a monospace font, disables spell check, and preserves spaces and special characters.
</Note>

### Links

Links connect to external URLs or workspace files.

<AccordionGroup>
  <Accordion title="Markdown syntax" icon="code" defaultOpen>
    Use the standard `[text](url)` format:

    ```markdown
    [Zditor](https://zditor.com)
    [Guide](./docs/guide.md)
    [Example](https://example.com "Optional title")
    ```
  </Accordion>

  <Accordion title="Auto-conversion" icon="sparkles">
    Once a full link expression is typed, Zditor converts it into a clickable link mark.
  </Accordion>

  <Accordion title="Interaction" icon="hand-pointer">
    In editing mode, clicking a link selects it for editing. In read-only or preview mode, clicking follows the link.
  </Accordion>

  <Accordion title="Link types" icon="link">
    Zditor supports external URLs, relative paths, absolute paths, and optional link titles.
  </Accordion>
</AccordionGroup>

<Warning>
  External links add safe `rel` attributes automatically to reduce security risks.
</Warning>

<Tip>
  Links are the foundation of the knowledge graph. See the [Knowledge Graph Guide](/en/knowledge-graph) for the graph-side behavior.
</Tip>

### Revision marks

Revision marks help track changes, suggestions, and review comments inside a document.

<AccordionGroup>
  <Accordion title="Core properties" icon="pen-to-square" defaultOpen>
    A revision mark can store:

    - `style` for the visual color
    - `mode` for the review mode
    - `title` for a short label
    - `advice` for an additional note

    Example:

    ```markdown
    This is <span class="revision">reviewed text</span>.
    ```
  </Accordion>

  <Accordion title="Shortcut" icon="keyboard">
    - macOS: `Cmd + Shift + R`
    - Windows/Linux: `Ctrl + Shift + R`
  </Accordion>

  <Accordion title="Interaction" icon="mouse-pointer">
    Clicking a revision selects the full range and reveals its stored details. Context menu actions can include accepting, rejecting, or editing the revision metadata.
  </Accordion>

  <Accordion title="Common uses" icon="lightbulb">
    Use revision marks for document review, collaborative writing, proofreading, and change tracking.
  </Accordion>

  <Accordion title="Color styles" icon="palette">
    Common styles include `teal`, `red`, `green`, `yellow`, and `blue`.
  </Accordion>
</AccordionGroup>

## Combining marks

### Allowed combinations

<AccordionGroup>
  <Accordion title="Most marks can be combined" icon="layer-group" defaultOpen>
    Common combinations include:

    ```markdown
    **_bold italic_**
    **==bold highlight==**
    [**bold link**](https://example.com)
    ```
  </Accordion>

  <Accordion title="Code stays exclusive" icon="ban">
    Inline code cannot be nested with other marks. For example:

    ```markdown
    `**not bold**`
    **`code wins`**
    ```
  </Accordion>

  <Accordion title="Nesting order matters" icon="arrows-down-to-line">
    Marks can be nested, but opening and closing pairs must remain well-formed. Crossing delimiters produces broken Markdown.
  </Accordion>
</AccordionGroup>

### Practical combinations

<CardGroup cols={2}>
  <Card title="Emphasis plus link" icon="link">
    Use bold or italic around a link when you want the link to stand out without changing its target.
  </Card>

  <Card title="Highlight plus bold" icon="highlighter">
    Combine highlight and bold for strong callouts such as definitions or conclusions.
  </Card>

  <Card title="Revision plus replacement" icon="pen-ruler">
    Use strikethrough for the old text and a revision mark for the proposed replacement.
  </Card>

  <Card title="Multi-emphasis" icon="stars">
    Combine several marks only when the readability payoff is clear.
  </Card>
</CardGroup>

<Warning>
  Avoid piling on too many marks. Two or three marks are usually enough for one text span.
</Warning>

## Shortcut reference

### macOS

| Mark | Shortcut | Action |
| --- | --- | --- |
| Bold | `Cmd + B` | Toggle bold |
| Italic | `Cmd + I` | Toggle italic |
| Strikethrough | `Cmd + Shift + X` | Toggle strikethrough |
| Highlight | `Cmd + Shift + H` | Toggle highlight |
| Inline code | `Cmd + E` | Toggle inline code |
| Revision | `Cmd + Shift + R` | Insert revision mark |

### Windows and Linux

| Mark | Shortcut | Action |
| --- | --- | --- |
| Bold | `Ctrl + B` | Toggle bold |
| Italic | `Ctrl + I` | Toggle italic |
| Strikethrough | `Ctrl + Shift + X` | Toggle strikethrough |
| Highlight | `Ctrl + Shift + H` | Toggle highlight |
| Inline code | `Ctrl + E` | Toggle inline code |
| Revision | `Ctrl + Shift + R` | Insert revision mark |

<Tip>
  These shortcuts act on the current selection. If no text is selected, they usually affect the text you type next.
</Tip>

## Input rule reference

| Mark | Input syntax | Example |
| --- | --- | --- |
| Bold | `**text**` + space | `**important**` |
| Italic | `_text_` + space | `_emphasis_` |
| Italic | `*text*` + space | `*note*` |
| Strikethrough | `~text~` + space | `~remove~` |
| Highlight | `==text==` + space | `==focus==` |
| Inline code | `` `text` `` + space | `` `code` `` |
| Link | `[text](url)` + space | `[Docs](./docs.md)` |

<Note>
  Input rules usually trigger on space, and in some cases on enter. They can also be disabled in editor settings if needed.
</Note>

  <Accordion title="Remove marks quickly" icon="eraser">

## FAQ

<AccordionGroup>
  <Accordion title="Why does underscore not always convert to italic?" icon="question">
    The conversion rule is strict to avoid false positives with names like `snake_case`. Use asterisks or apply italic through the shortcut when needed.
  </Accordion>

  <Accordion title="How do I show formatted text inside code?" icon="question">
    You generally should not. Inline code renders as plain code text by design and ignores other marks.
  </Accordion>

  <Accordion title="Can links be wrapped in other marks?" icon="question">
    Yes, except inside inline code. Bold links and italic links are both supported.
  </Accordion>

  <Accordion title="Can I span marks across multiple paragraphs?" icon="question">
    Marks are inline constructs, so they normally stay within a paragraph. For larger structural emphasis, use block-level elements instead.
  </Accordion>

  <Accordion title="Are revision marks stored in Markdown?" icon="question">
    Yes. Revision data is stored as HTML markup embedded in the Markdown content, so it can be exported and preserved.
  </Accordion>

  <Accordion title="Can I customize the shortcuts?" icon="question">
    That depends on the app version and editor settings. Some bindings are configurable, while others are fixed.
  </Accordion>
</AccordionGroup>

## Best practices

<Steps>
  <Step title="Keep formatting restrained">
    Use marks to support meaning, not to overwhelm the text.
  </Step>

  <Step title="Define a consistent convention">
    For example: bold for key terms, italic for foreign words, highlight for definitions, code for technical tokens, and revision marks for review feedback.
  </Step>

  <Step title="Use shortcuts heavily">
    The keyboard workflow is much faster than manual symbol editing once the shortcuts become muscle memory.
  </Step>

  <Step title="Use auto-conversion where it helps">
    Input rules speed up repetitive formatting, but disable rules that often misfire in your content.
  </Step>

  <Step title="Protect readability">
    Test how your marks look under different themes and avoid flashy combinations that distract from the content.
  </Step>
</Steps>
