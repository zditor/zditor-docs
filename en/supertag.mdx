SuperTag lets Zditor treat Markdown files as structured records and render references to those records as tag-like nodes inside documents.

## Overview

<CardGroup cols={2}>
  <Card title="Structured records" icon="database">
    Store reusable metadata in frontmatter while keeping the body as normal Markdown.
  </Card>

  <Card title="Shared class" icon="table">
    Group many files into one logical table by giving them the same `class` value.
  </Card>

  <Card title="Inline references" icon="link">
    Insert a record into another document with `mode=supertag`.
  </Card>

  <Card title="Card layout fields" icon="table">
    Use fields such as `col`, `row`, `cover`, and `warm` to shape visual presentation.
  </Card>
</CardGroup>

## Core Concepts

<AccordionGroup>
  <Accordion title="A record is still a Markdown file" icon="file" defaultOpen>
    A SuperTag record is a normal Markdown document with typed frontmatter.

    ```yaml
    ---
    title:
      type: text
      label: Title
      value: "The Godfather"
    director:
      type: text
      label: Director
      value: "Francis Ford Coppola"
    year:
      type: number
      label: Year
      value: 1972
    class:
      type: text
      label: class
      value: "ClassicMovieEnglish"
    ---
    ```
  </Accordion>

  <Accordion title="The class field groups records" icon="table">
    Files that share the same `class` value belong to the same logical table.

    ```yaml
    class:
      type: text
      label: class
      value: "ClassicMovieEnglish"
    ```

    This is what lets Zditor aggregate many Markdown files into one structured collection.
  </Accordion>

  <Accordion title="col and row shape card layout" icon="table">
    Layout fields control which metadata is emphasized on document cards.

    ```yaml
    col:
      type: array
      label: Col
      value: [year, title, director]
    row:
      type: array
      label: Row
      value: [genre, rank, score, watched]
    ```

    - `col` is for the main vertical information area
    - `row` is for the supporting or bottom information band
  </Accordion>
</AccordionGroup>

## Reference Syntax

Use a normal Markdown link with `mode=supertag`:

```markdown
[title](src|mode=supertag)
```

<Note>
  The minimal product syntax is `[title](src|mode=supertag)`. In real documents, `src` is usually the target file path.
</Note>

## Common Field Types

| Type | Typical use |
| --- | --- |
| `text` | title, author, country, notes |
| `number` | year, score, rank |
| `date` | due dates, watch dates, created dates |
| `checkbox` | done state, warm theme, featured state |
| `multiselect` | genre, tags, categories |
| `array` | cast lists, layout field lists |
| `progress` | score-like or progress-like values |
| `asset` | covers and resource paths |

## Recommended Workflow

<Steps>
  <Step title="Copy an existing record">
    Start from a file that already has the schema you want.
  </Step>

  <Step title="Keep the field structure and class">
    Reuse the same keys, types, and `class` value so the record stays in the same collection.
  </Step>

  <Step title="Replace concrete values">
    Update the title, dates, cover, and body content to match the new record.
  </Step>

  <Step title="Backfill new shared fields later">
    If the whole class needs another field, add it consistently across the collection.
  </Step>
</Steps>

<Tip>
  SuperTag works best when field names stay stable over time. Avoid casual renames once a class is already in use.
</Tip>
